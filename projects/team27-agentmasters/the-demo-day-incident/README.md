# The Demo Day Incident

추리 게임 프로젝트입니다. 플레이어는 데모데이 현장에서 발생한 사건을 조사하고, 단서와 증언을 조합해 진실을 추론합니다.

## 레포지토리 구성

이 부모 레포는 프로젝트의 공통 문서와 설계 정보를 관리합니다.

| 레포 | 역할 |
| --- | --- |
| `the-demo-day-incident` | 기획, 설계, 회의록, 공통 협업 규칙 관리 |
| `frontend` | 클라이언트 화면, 사용자 인터랙션, 프론트엔드 상태 관리 |
| `backend` | API, 데이터 모델, 게임 진행 로직, 서버 인프라 |

프론트엔드와 백엔드 구현 작업은 각각의 레포에서 진행하고, 설계 변경이나 공통 의사결정은 이 부모 레포에 문서화합니다.

## 협업 규칙

### 작업 흐름

1. 작업을 시작하기 전에 GitHub Issue를 먼저 생성합니다.
2. Issue 번호를 기준으로 작업 브랜치를 생성합니다.
3. 작업이 끝나면 해당 작업이 속한 레포에 Pull Request를 생성합니다.
4. 설계, 정책, 공통 규칙 변경은 이 부모 레포에 PR을 생성합니다.
5. 프론트엔드 구현은 `frontend` 레포에 PR을 생성합니다.
6. 백엔드 구현은 `backend` 레포에 PR을 생성합니다.

### 브랜치 네이밍

브랜치 이름은 아래 형식을 따릅니다.

```text
<prefix>/<issue-num>
```

예시:

```text
feat/12
fix/24
docs/31
refactor/42
```

권장 prefix:

| Prefix | 용도 |
| --- | --- |
| `feat` | 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `refactor` | 동작 변경 없는 코드 구조 개선 |
| `test` | 테스트 추가 또는 수정 |
| `chore` | 설정, 빌드, 기타 작업 |

### 커밋 컨벤션

커밋 메시지는 Conventional Commits 형식을 사용하되, 설명은 한국어로 작성합니다.

```text
<type>: <한국어 요약>
```

예시:

```text
feat: 단서 목록 화면 추가
fix: 증거 선택 시 잘못된 상태가 저장되는 문제 수정
docs: 협업 규칙 문서화
refactor: 게임 진행 상태 계산 로직 분리
test: 추리 결과 판정 테스트 추가
chore: 린트 설정 추가
```

본문이 필요한 경우에는 무엇을 바꿨는지보다 왜 바꿨는지를 중심으로 한국어로 작성합니다.

### PR 컨벤션

PR 제목은 커밋 컨벤션과 동일하게 작성합니다.

```text
<type>: <한국어 요약>
```

PR 본문에는 아래 항목을 포함합니다.

```markdown
## 작업 내용
- 

## 관련 이슈
- Closes #

## 확인 방법
- 

## 참고 사항
- 
```

## 문서 관리 규칙

공통 설계와 의사결정은 이 부모 레포에서 관리합니다.

- 게임 기획, 시나리오 구조, 핵심 규칙
- API 계약과 데이터 모델 초안
- 프론트엔드/백엔드 간 인터페이스 합의
- 팀 협업 규칙
- 회의록과 결정 사항

구현 세부사항은 각 레포의 README와 코드 문서에 작성합니다.
