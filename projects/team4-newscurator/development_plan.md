# 뉴스 요약 큐레이터 에이전트 개발 계획

## 1. 프로젝트 목표

사용자가 선호 언론사, 관심 분야, 직접 키워드를 선택하면 최신 뉴스를 수집하고, 중복 기사를 줄인 뒤, 핵심 내용만 짧고 중립적으로 요약해 제공하는 개인 맞춤형 뉴스 브리핑 서비스를 개발한다.

핵심 가치는 사용자가 여러 언론사와 기사 목록을 직접 탐색하지 않아도 관심 분야의 주요 이슈를 빠르게 파악하도록 돕는 것이다.

## 2. 완성형 MVP 범위

완성형 MVP는 발표용 시연을 넘어 실제 사용자가 조건을 바꿔 반복적으로 브리핑을 생성할 수 있는 수준을 목표로 한다.

- 공개 RSS가 확인된 언론사 선택 및 검색
- 관심 분야 선택
- 직접 키워드 입력
- 추천 프리셋
- 공개 RSS 피드 기반 실제 뉴스 수집
- 기사 필터링 및 중복 제거
- Upstage Solar 기반 기사 요약
- 기사 우선순위 점수 및 선정 기준 표시
- 요약 결과 목록 출력
- 원문 기사 링크 제공
- 최근 브리핑 기록 저장 및 다시 보기
- 브리핑 생성 기준 표시
- 기본 예외 처리 및 샘플 데이터 fallback

MVP에서는 사용자 로그인, 장기 개인화 학습, 뉴스 진위 판단, 정치적 의견 제공, 투자 추천, 기사 작성 기능은 제외한다.

## 3. 권장 기술 구조

기획서의 기술 스택을 살리되 구현 복잡도를 낮추기 위해 다음 구조를 권장한다.

- Frontend: React + Vite, Node.js
- Backend: Python FastAPI
- AI 요약: Upstage Solar API
- 뉴스 수집: 공개 RSS 우선, NewsAPI 선택 fallback
- Database: SQLite
- 협업/버전관리: GitHub

프론트엔드는 사용자의 선택값 입력과 결과 표시를 담당한다. 백엔드는 뉴스 수집, 필터링, 중복 제거, 요약 요청, 결과 저장을 담당한다.

## 4. 주요 사용자 흐름

1. 사용자가 서비스에 접속한다.
2. 사용자가 언론사를 선택한다.
3. 사용자가 관심 분야와 직접 키워드를 선택한다.
4. 사용자가 뉴스 조회를 요청한다.
5. 백엔드가 공개 RSS에서 관련 최신 뉴스를 수집한다.
6. 백엔드가 기사 제목, 설명, URL, 출처, 발행일을 기준으로 필터링한다.
7. 유사 제목과 동일 URL을 기준으로 중복 기사를 제거한다.
8. 키워드 관련성과 최신성을 기준으로 기사 우선순위를 계산한다.
9. Upstage Solar API로 핵심 요약을 생성한다.
10. 중요도와 최신순을 기준으로 결과를 정렬한다.
11. 사용자가 요약, 출처, 발행일, 우선순위, 원문 링크를 확인한다.

## 5. 기능 상세 계획

### 5.1 언론사 및 관심 분야 선택

- 기본 언론사 목록을 제공한다.
- 관심 분야는 IT, AI, 경제, 스타트업, 주식시장 등 기획서에 나온 페르소나 중심으로 시작한다.
- 사용자가 아무 항목도 선택하지 않으면 선택 요청 메시지를 표시한다.

### 5.2 뉴스 수집

- NewsAPI 요청 파라미터를 백엔드에서 구성한다.
- 선택한 언론사, 관심 키워드, 기간을 기준으로 기사를 가져온다.
- MVP에서는 최근 1일 또는 최근 7일 범위를 우선 지원한다.
- API 오류가 발생하면 재시도 안내 메시지를 반환한다.

### 5.3 기사 필터링 및 중복 제거

- 제목, 설명, 출처, URL, 발행일을 표준 필드로 정리한다.
- 동일 URL은 중복으로 제거한다.
- 제목이 매우 유사한 기사는 한 건만 남긴다.
- 관심 분야 키워드와 관련성이 낮은 기사는 제외한다.
- 제목·본문 키워드 매칭과 발행 시점을 기준으로 우선순위 점수를 계산한다.

### 5.4 Upstage Solar 요약

- 원문 기반, 중립적, 간결한 브리핑 톤을 유지한다.
- 기사별 요약은 2~4문장으로 제한한다.
- 여러 기사에서 공통으로 나타나는 이슈는 “공통 핵심 이슈”로 별도 정리한다.
- 정치적 해석, 투자 추천, 진위 판단은 하지 않는다.

권장 출력 형식:

```json
{
  "briefing_title": "오늘의 AI 뉴스 브리핑",
  "common_topics": ["공통 이슈 1", "공통 이슈 2"],
  "articles": [
    {
      "title": "기사 제목",
      "source": "언론사",
      "published_at": "발행일",
      "summary": "핵심 요약",
      "why_it_matters": "중요한 이유",
      "url": "원문 링크"
    }
  ]
}
```

### 5.5 결과 화면

- 상단에 선택한 언론사, 관심 분야, 조회 기간을 표시한다.
- 핵심 브리핑 요약을 먼저 보여준다.
- 기사 목록은 카드 또는 리스트 형태로 표시한다.
- 각 기사에는 제목, 언론사, 발행일, 요약, 원문 링크를 포함한다.
- 각 기사에는 우선순위 배지와 선정 기준을 함께 표시한다.
- 최근 생성한 브리핑을 저장하고 다시 열 수 있다.
- 관련 뉴스가 없으면 “관련 뉴스가 없습니다.”를 표시한다.

## 6. API 설계 초안

### `GET /health`

서버 상태 확인용 API.

### `GET /sources`

지원 언론사 목록을 반환한다.

### `GET /topics`

지원 관심 분야 목록을 반환한다.

### `POST /briefings`

사용자 선택값을 받아 뉴스 수집과 요약을 실행한다.

요청 예시:

```json
{
  "sources": ["yonhap", "mk", "hankyung"],
  "topics": ["ai", "it"],
  "custom_keywords": ["로봇", "바이오"],
  "date_range": "7d",
  "limit": 5
}
```

응답 예시:

```json
{
  "status": "ok",
  "briefing": {
    "title": "AI 및 IT 뉴스 브리핑",
    "common_topics": [],
    "articles": []
  }
}
```

### `GET /briefings/history`

최근 생성한 브리핑 기록 목록을 반환한다.

### `GET /briefings/{briefing_id}`

저장된 브리핑 상세 내용을 반환한다.

## 7. 데이터베이스 설계 초안

SQLite는 MVP 기준으로 캐시와 기록 저장에 사용한다.

### `articles`

- `id`
- `title`
- `source`
- `url`
- `published_at`
- `description`
- `topic`
- `created_at`

### `briefings`

- `id`
- `sources`
- `topics`
- `date_range`
- `result_json`
- `created_at`

### `api_logs`

- `id`
- `provider`
- `status`
- `message`
- `created_at`

## 8. 팀 역할 배정안

역할은 실제 팀 선호에 맞춰 조정할 수 있다.

- PM/기획: 요구사항 정리, 일정 관리, 사용자 흐름 구성
- Frontend: 화면 구성, 선택 UI, 결과 리스트 구현
- Backend: FastAPI 서버, API 라우팅, SQLite 연동
- Data/News: 공개 RSS 연동, 기사 필터링, 중복 제거
- AI/QA: 요약 프롬프트, 예외 처리, 테스트, 운영 흐름 검증

5명 팀 기준으로는 참여자별로 위 역할을 하나씩 맡고, 통합 테스트와 발표 준비는 전원이 함께 진행하는 방식이 적합하다.

## 9. 2주 개발 로드맵

### 1일차: 프로젝트 세팅

- GitHub 저장소 구조 생성
- Frontend/Backend 기본 프로젝트 생성
- `.env.example` 작성
- RSS 수집 방식 정리
- Upstage API 키 관리 방식 정리
- README 초안 작성

### 2~3일차: 뉴스 수집 기능

- 공개 RSS 요청 모듈 구현
- 언론사/관심 분야 매핑 구현
- 기사 데이터 표준화
- API 오류 및 기사 없음 처리

### 4~5일차: 필터링 및 중복 제거

- 동일 URL 중복 제거
- 유사 제목 중복 제거
- 관심 분야 키워드 필터링
- 최신순/관련도 기준 정렬

### 6~7일차: Upstage Solar 요약 기능

- 기사별 요약 프롬프트 작성
- 공통 이슈 추출 프롬프트 작성
- JSON 응답 파싱 및 실패 처리
- 요약 실패 시 원문 링크 중심 결과 반환

### 8~9일차: UI 구현

- 언론사 선택 UI
- 관심 분야 선택 UI
- 추천 프리셋 버튼
- 언론사 검색 및 전체 선택/해제
- 직접 키워드 입력
- 조회 버튼 및 로딩 상태
- 요약 결과 리스트
- 원문 링크 연결

### 10~11일차: 통합 및 예외 처리

- Frontend와 Backend 연동
- 사용자가 선택하지 않은 경우 처리
- API 오류 처리
- 요약 실패 처리
- 관련 뉴스 없음 처리

### 12일차: 테스트

- 뉴스 수집 단위 테스트
- 중복 제거 테스트
- 요약 응답 형식 테스트
- 주요 사용자 시나리오 테스트

### 13일차: 최종 검증 준비

- 프리셋 1: 오늘 IT/경제 뉴스 요약
- 프리셋 2: AI·반도체 주간 뉴스 요약

### 14일차: 최종 정리

- 실행 방법 문서화
- 최종 사용자 흐름 점검
- 최종 버그 수정

## 10. 테스트 계획

- 사용자가 언론사를 선택하지 않았을 때 안내 메시지가 나오는지 확인한다.
- 사용자가 관심 분야를 선택하지 않았을 때 안내 메시지가 나오는지 확인한다.
- RSS 수집 실패 시 안내 또는 fallback이 동작하는지 확인한다.
- 관련 뉴스가 없을 때 “관련 뉴스가 없습니다.”가 나오는지 확인한다.
- 요약 실패 시 원문 기사 링크가 제공되는지 확인한다.
- 결과가 제목, 출처, 요약, 링크를 모두 포함하는지 확인한다.
- 요약 내용이 정치적 의견이나 투자 추천처럼 제외된 기능을 수행하지 않는지 확인한다.

## 11. 주요 리스크 및 대응

- 공개 RSS 응답 실패: 샘플 fallback, 조회 건수 제한, 실패 피드 안내
- 유료 기사 전문 접근 불가: 제목, 설명, 공개 본문 중심 요약으로 제한
- 요약 품질 편차: JSON 출력 형식 고정, 프롬프트 테스트 반복
- 정치적 편향 우려: 원문 기반 요약, 의견/판단 금지 문구 삽입
- 실시간성 한계: “최신”의 기준을 RSS 발행일 기준으로 명확히 표시
- 외부 API 장애: 샘플 fallback과 로컬 요약으로 핵심 흐름 유지

## 12. 완료 기준

프로젝트는 다음 조건을 만족하면 MVP 완료로 본다.

- 사용자가 언론사, 관심 분야, 직접 키워드를 선택할 수 있다.
- 선택값 기반으로 뉴스 데이터를 수집할 수 있다.
- 관련 기사 5건 내외를 요약해 보여줄 수 있다.
- 각 결과에 원문 링크가 포함된다.
- 기사 없음, API 오류, 요약 실패, 선택 누락 예외가 처리된다.
- 추천 프리셋과 직접 조건 입력 흐름을 안정적으로 사용할 수 있다.
